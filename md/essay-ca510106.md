我在 GitHub 上搭了一个个人网站。HTML/CSS/JS，Flask 做无头 CMS，GitHub Pages 部署。技术栈朴素到有点复古，但省心，是真的省心。

写点东西，贴几张照片而已。

直到有一天，我想往上面放一些不想让所有人看到的东西。

不是要建什么暗网，只是想给某些文章加一把锁——你自己有钥匙，想看了打开，不想看的时候它就在那里加密着。

访客看到的是标题和摘要，点进去只有一个密码框。输入对了，文章才出现。

>听起来很简单对吧。但做着做着我才发现，这件事比想象中有意思得多。

## 一、最直接的做法，也是最不行的

最**直觉**的方案：服务器存密码，用户输入后服务端校验，通过了就返回文章内容。Flask 后端，session 鉴权，几行代码的事。

问题是——我的网站是**静态**的。

GitHub Pages 上跑的是一堆已经生成好的 HTML 文件。没有服务器，没有 Flask，没有 session。

python manage.py build 在 CI 上跑完，把生成好的静态文件推到 Pages 上，整个部署就完成了。

加密文章必须在构建的时候就生成好。

>要么把解密放在服务端，要么把解密放在客户端。

如果你把密码放在 CI 上，**那密码就不再是你的密码了**。

它变成了 GitHub 服务器上一段可以被任何人读取的环境变量——只要他们有足够的权限，或者足够的好奇心。

所以第一个方向被毙掉了。密码不能碰服务器。

## 二、把密码锁在浏览器里

那如果反过来呢——构建的时候文章已经是加密状态，直接嵌到 HTML 里，读者打开页面，输入密码，浏览器自己在**本地**解密？

>这条路的关键在于：构建时不需要密码。

文章的 .md 源文件在本地用 Fernet（AES-128-CBC + HMAC-SHA256）加密后提交到 Git——这一步用的是 Python 的 cryptography 库，只在我自己的电脑上跑，不上 CI，也不进浏览器。

CI 构建时读到的只是密文。

>它不需要知道密码是什么，只需要检测到"这是加密内容"，然后原样塞进 HTML 模板里。

读者打开页面，看到密码框。

输入密码 → PBKDF2-SHA256 派生密钥 → Web Crypto API 在浏览器里解密 → 得到 Markdown 原文 → marked.js 渲染成 HTML → DOMPurify 净化 → 显示。

>密码从始至终只存在于你的大脑和你面前的浏览器里。它从来不曾出现在任何一台服务器、任何一段 CI 日志、任何一个 Git 提交中。

## 三、Fernet 的格式刚好是答案

选 Fernet 不是因为它最安全，是因为它的格式刚好适合这个场景。

一个 Fernet token 的结构长这样：version(1byte) + timestamp(8bytes) + IV(16bytes) + ciphertext + HMAC(32bytes)。所有东西打包在一段 base64 字符串里，一个文件就是一条完整可验证的加密信息。

构建时读取 .md 文件，检查第一个字节是不是 0x02（版本号）：是 → 加密内容，直接透传；不是 → 明文，走原来的渲染流程。

检测逻辑只有四行代码，不需要任何外部依赖。

>客户端解密的时候，Web Crypto API 刚好有我们需要的所有部件：PBKDF2、HMAC-SHA256、AES-CBC。不需要引入任何第三方加密库，浏览器原生支持。

解密这一步是真正零依赖的——你不需要信任一个 npm 包作者没有在某个凌晨的 commit 里动过手脚。但零依赖不等于整个系统都没有外部代码。真正的第三方依赖，出现在下一步。

## 四、渲染与净化

解密只是第一步。解密出来的是 Markdown 原文，还需要渲染成 HTML。

之前服务端用 Python 的 markdown 库渲染。搬到客户端后，换成了 marked.js——一个纯 JS 的 Markdown 解析器，支持 GFM 扩展，20KB 左右，通过 CDN 加载。

但 Markdown 解析器允许原始 HTML，意味着一段包含 <script> 标签的恶意内容可能被注入到页面里。虽然加密保证了外人改不了密文，但自己写的内容也可能无意识地包含危险片段。

DOMPurify 解决了这个问题——同样是 CDN 加载，在 marked.parse() 之后再过一遍净化，白名单过滤所有非安全标签。

>加密保证的是"外面的人进不来"，净化保证的是"里面的东西不会伤到自己"。

整套系统一共用了三处依赖，而不是零处——本地加密用 cryptography 库，渲染和净化各用一个 CDN 脚本。

真正零依赖的只有浏览器里那一步解密。加密的核心是干净的，外围的渲染和净化不是，这两件事不该被混为一谈。

五、密钥派生——PBKDF2 的 100,000 次迭代

从用户输入的密码到 Fernet 需要的 256 位密钥，中间要经过一次密钥派生。

PBKDF2-SHA256，100,000 次迭代，16 字节**随机盐**。

每次加密都生成新的盐。

>所以同样的密码加密同样的内容，每次出来的密文都不一样。

盐存在密文的第二个字节到第十七个字节之间，解密的时候从密文里取出来用。

100,000 次迭代在我自己的笔记本浏览器里实测大约是几十毫秒——感觉不到延迟，但对暴力破解来说足够慢。

OWASP 建议 600,000 次。但 100,000 次在浏览器里是一个合理的折中：安全性足够，用户体验不会受影响。

但说实话我的网站其实并不会有人来暴力破解hh，纯属职业病式的过度设计而已。

## 六、CI 上的表现

改这套系统之前，CI 构建日志里最常见的是这样一行：

WARNING: orphan ciphertext detected — essays/essay-032f74a6.html rendered blank

加密内容检测不到，模板拿到的是一段既不是合法 Markdown、也解不开的字符串，页面直接空白。

改完之后，同一篇文章的构建日志变成了：

✓ essays/essay-032f74a6.html

不再有 orphan ciphertext 警告，不再有空白页。

加密文章的 HTML 正常生成，密码门正常出现，_encryptedBody 和 _encryptedIsMd 两个变量嵌在模板里，等着正确的人输入正确的密码。

>一个东西修好了的标志不是它不再出错，是你回头看的时候，觉得它本来就应该这样。

## 七、你真的需要加密吗

做完之后我停了一段时间，回头看这套东西。

加密系统本身很干净，密码不碰服务器，解密全在浏览器完成。技术上是成立的。

但有没有可能，这件事从一开始就不需要这么复杂？

有些东西，你不写出来就是了。

有些东西，写出来了，但放在一个不公开的地方——一个本地的文件夹，一个只有你自己知道的路径，其实也就够了。

>不是所有想藏起来的东西都需要一把锁。有时候墙就够了。

但如果你像我一样，就是想把它放在网上，放在一个你自己也随时能看到的地方，同时又不想让所有人都能看到——那这把锁就值得。

不是为了安全，是为了**选择**。

你选择哪些东西公开，哪些东西需要一把钥匙。

>钥匙在你手里，你想给谁就给谁。这本身就是一种表达。
