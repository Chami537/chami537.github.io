# 全项目优化设计

日期：2026-07-15

## 目标

在不改变网站内容模型、管理面板工作流和公开页面视觉语义的前提下，先修复已经复现的安全、数据完整性和 CI 可靠性缺陷，再收紧后端数据边界、减少重复实现和无效兼容层。优化完成后，所有现有测试继续通过，并为每个已确认缺陷增加能够先失败、修复后通过的回归测试。

## 当前基线

- 134 个可审查的源码或配置文件
- 70 个 Flask 端点
- `python -m pytest -q`：192 passed
- 45 个 JavaScript 文件通过 `node --check`
- Python 编译检查和 `git diff --check` 通过
- TokenSave 项目索引可用
- Graphify 图谱可用，但实现完成后需要基于当前提交重新生成

## 范围与原则

本次优化分为两个有序阶段。第一阶段只处理已有复现证据的缺陷，不趁机改版或扩展功能；第二阶段只清理与这些数据流、服务边界直接相关的架构问题。每个阶段都保持现有 HTTP 成功响应、JSON 结构、数据文件格式和前端交互兼容。

所有输入在进入渲染、文件写入或外部命令边界前完成验证。无法证明输入安全或数据完整时应失败并保留原状态，而不是静默纠正、清空或覆盖。重构优先复用现有模块，不引入新框架或第三方运行时依赖。

## 阶段一：安全与可靠性修复

### 1. 公开页面输出安全

公开首页中进入 HTML 文本和双引号属性的动态值使用上下文正确的转义，至少覆盖 `&`、`<`、`>`、`"` 和 `'`。照片及故事日期不得以未转义字符串拼接进 `innerHTML`；能使用 DOM 属性或 `textContent` 的位置优先使用 DOM API。

Markdown 链接先解码 HTML 字符引用、去除会影响协议识别的控制符和空白，再解析规范化协议。只允许既有业务需要的安全协议和相对链接；`javascript:` 及其大小写、实体编码、控制符变体全部被拒绝。显示文本仍按现有 Markdown 行为渲染。

### 2. CSRF 可信来源

可信来源由服务端配置和固定的本地开发来源决定，不再从 `request.host` 推导。请求的 `Origin` 或 `Referer` 必须与可信来源精确匹配；攻击者控制 `Host` 头不能扩大信任集合。测试模式继续支持测试客户端，但必须显式配置，不使用 Host 回退。

### 3. 照片同步与损坏数据保护

照片同步分别处理“已有照片”和“新增照片”：已有照片保留非派生字段，如 `tags`、`size` 和人工维护元数据；新增照片从 EXIF 和文件属性生成完整记录。日期存在与否不再决定记录是否被加入。

读取 `photos.json` 时若发生解析或结构错误，同步命令应以非零状态停止，保留原文件且不写出空数组。所有持久化继续使用原子替换；临时图像输出也必须在数据验证失败时停止提交。

### 4. 随笔密码生命周期

slug 改名与随笔元数据更新属于同一业务操作。改名成功时密码映射键同步迁移，目标键冲突时拒绝操作并保持两个文件不变；删除随笔时清理对应密码键。无密码随笔不创建空记录。该行为由集中助手负责，路由不直接散写密码文件。

### 5. README 与登录输入验证

README 更新在打开目标文件前确认 JSON 对象和 `content` 字符串类型；非法请求返回 400，现有 README 保持原样。合法内容通过临时文件和原子替换写入，避免中途异常留下空文件。

登录密码只接受字符串。数字、空值、数组和对象统一返回认证失败或 400，不传给 `hmac.compare_digest`，也不产生 500。成功和普通错误密码的既有行为保持不变。

### 6. 缓存清单与 CI

缓存刷新目标从 `index.html` 和 `admin.html` 中实际引用的本地 CSS/JS 资源派生，避免手工清单漏掉 `admin-api.js`、`admin-ui.js` 或未来新增资源。只更新已有本地资源查询参数，不触碰外部 URL。

CI 顺序调整为：安装依赖、运行单元/接口测试、执行全站构建、验证构建产物、运行 Chromium 冒烟。产物测试不得因目标文件不存在而跳过断言；缺少预期产物即失败。Git 破坏性操作仍在测试模式隔离，同时补充命令失败路径的契约测试。

## 阶段二：架构瘦身

### 1. 存储组合根

消除 `backend.data -> backend.storage -> backend.storage.repository -> backend.data` 的延迟导入环。路径常量、JSON 编解码和原子文件操作位于底层模块；Repository 只依赖这些底层接口；应用组装在更高层完成。过渡期间保留外部仍在使用的 `load_json` 和 `atomic_write_json` 导入位置，但实现单向委托，并用测试锁定兼容性。

### 2. 单一照片数据所有者

将 `photos.json` 的规范化、读取、合并和写入集中到一个照片仓储或领域服务。Flask 路由、同步工具和 SSG 只调用公开接口，不各自实现合并规则。图像文件处理仍属于工具层，数据记录提交由统一所有者完成。

### 3. 随笔应用服务

路由不再直接依赖 SSG 私有函数。建立公开的随笔应用服务，封装元数据更新、slug 迁移、密码映射、Markdown/HTML 同步和公开目录刷新。SSG 保留纯生成职责，路由只负责 HTTP 验证与响应转换。

### 4. 重复与死代码

公开页和管理预览共享同一份语法高亮规则或生成数据，避免两套约 90 行逻辑继续漂移。只删除已通过全仓引用检查和测试证明无调用的包装函数，包括未使用的 index CRUD 包装、Repository 查询包装、星标抓取包装及已被新服务替代的 SSG 兼容层。不会为了减少行数合并职责不同的模块。

## 数据流与错误约定

写操作采用统一顺序：HTTP/CLI 输入验证 -> 规范化 -> 应用服务校验跨文件约束 -> Repository 原子提交 -> 派生文件刷新 -> 返回结果。验证失败返回明确的 4xx 或 CLI 非零退出码；磁盘、构建或 Git 失败返回 5xx/非零退出码并记录原因。任何失败都不得把有效文件替换为空值或部分状态。

跨文件更新无法使用数据库事务，因此在写入前完成全部验证并准备新内容，再以可恢复顺序原子替换；若第二个文件替换失败，恢复第一个文件的原内容并让操作失败。对应故障注入测试覆盖回滚路径。

## 测试与验收

每个确认缺陷遵循 RED -> GREEN：先加入能稳定复现问题的最小测试，确认旧实现失败，再实现修复。架构调整在行为测试保护下分小步完成。

最终验收包括：

1. 安全与数据缺陷的定向回归测试。
2. `python -m pytest -q` 全量测试。
3. 强制执行 Chromium 浏览器冒烟，不把断言异常转换为 skip。
4. 全部 JavaScript 文件执行 `node --check`。
5. Python 源码编译检查。
6. `python manage.py build` 及构建后产物验证。
7. `git diff --check`。
8. `graphify update .`，确认图谱基于最终代码。
9. 检查构建产生的缓存参数和 JSON 差异，只保留本次优化必需的变更。

## 交付边界

实现提交按“安全与数据完整性”“服务边界与清理”拆分，使回归定位和回滚清晰。不会修改用户内容、引入前端框架、改变部署分支、自动推送远端或发布站点；推送与发布仍需单独指令。
